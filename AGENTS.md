# AGENTS.md — 執筆・改訂ルール

このリポジトリは OSS のソースコード読解ドキュメントを日本語で蓄積する。
LLM が章を加筆・改訂するときは、一貫性を保つため以下を必ず守る。
（新規 OSS ドキュメントの立ち上げ手順や背景知識は `.wiki/methodology.md` を参照する。）

## 1. 使用スキル（必須）

- 執筆・推敲の前に必ず `japanese-tech-writing` スキルを読み、その規範に従う。
  一文一行／パラグラフライティング／ダッシュ（— ― ——）と中黒（・）を地の文で使わない／用語の初出は太字・以後は「」／LLM っぽい空句や過剰演出を書かない／推量を機械的に断定へ変えない。

## 2. リポジトリ構成

- OSS ごとに専用ディレクトリ（例 `rocksdb/`）を作り、その直下 `README.md` を目次とする。
- 章は部ごとのサブディレクトリに連番で置く（例 `rocksdb/part03-sst/15-block-based-table-builder.md`）。
- 章は基礎から順に積み上がる構成にし、前提章へのリンクで依存を示す。

### 目次 `README.md` の体裁（全 OSS 共通）

既存の RocksDB / Valkey / PostgreSQL / MySQL を次の並びに統一している。新しい OSS もこれに合わせる。

```text
# <OSS名> ソースコードリーディング

<概要1文：<OSS名>（[owner/repo](URL)）のソースコードを読み解き、…を日本語で解説するドキュメントである。>

- **対象バージョン**：X.Y.Z（コード引用はすべて [`<tag>` タグ](URL/tree/<tag>)に固定）
- **想定読者**：…
- **読み方**：…（1文）

<コード引用方式を説明する1文。必要なら各書固有の方針を1文だけ添えてよい。>

## 第0部　…

1. [章名](part00-…/01-….md)
2. …

## 第1部　…

…

---

> <脚注（執筆状況やタグの注記）。>
```

- 部見出しは `## 第N部` を直接置く。`## 目次` ラッパーや `### ` の部見出しは使わない。
- 章番号は部をまたいで通し連番にする（markdownlint の MD029 は無効化済み）。
- 末尾の脚注（`>` 引用）の前に区切り線 `---` を必ず置く。
- 導入は箇条書き（対象バージョン／想定読者／読み方）に統一する。地の文の段落で長く書かない。
- README でも一文一行を守る（箇条書きの値・脚注も1行1文。括弧内に句点を埋め込まない）。

## 3. コード引用（最重要）

- 引用は必ず「リンク行」＋「コードブロック」の2点セットで示す。
- リンクは**バージョン固定**の GitHub URL：`https://github.com/<owner>/<repo>/blob/<tag>/<rel>#L<開始>-L<終了>`。
  バージョンタグ（例 `v11.1.1`）に固定し、行番号ズレを防ぐ。リンクテキストは「リポジトリ相対パス L範囲」。
- 行番号と内容は**実ソースを読んで厳密に確認**する。記憶で書かない（バージョン間で API・行番号は動く）。
- **ローカル絶対パスを成果物に書かない**。引用は GitHub リンクで行う。
- コードは改変しない。長い場合は `// ... (中略) ...` で省く。コメントは原文（英語）のままでよい。

## 4. 章の構成テンプレート

```
# 第N章 タイトル
> 本章で読むソース（GitHub リンクの箇条書き）
## この章の狙い   ## 前提
（本文の節：目的 → コード引用 → 処理の流れ → 高速化・最適化の工夫）
## まとめ   ## 関連する章
```

## 5. 用語の統一

- 訳語・カタカナ表記は OSS ごとに統一する。新語を勝手に増やさない。クラス名・関数名はコードと同じ綴りをコードフォント（`...`）で示す。
- 既存章で使われている表記に合わせる（例 RocksDB では「カラムファミリー」「イテレータ」「シーケンス番号」「トゥームストーン」、MemTable・SST・WAL・Block Cache 等は英字）。
- **リポジトリ共通**：wrapper（ある対象を薄く覆う層）の訳語は「ラッパー」に統一する。「ラッパ」（長音なし）と書かない。名詞として「包み」と書かない（動詞の連用形「包み」は対象外）。
- **リポジトリ共通**：プロセスやスレッドが待機してブロックする動作（sleep。条件変数・ミューテックス・`TimedWait` などで CPU を手放して待つこと）の訳語は「スリープ（する）」に統一する。「眠る」「眠り」「休止」「休眠」と書かない（一般語としての睡眠の比喩ではなく、計算機でスレッドが待機・ブロックする箇所が対象）。

## 6. 内容の方針

- 「何のために、どういう処理をするか」を必ず説明し、コードの逐語訳に終始しない。
- 各章で**高速化・最適化の工夫を最低1つ、機構レベルで**説明する（なぜ速いのかを一文で）。
- コードから読み取れる事実と、設計意図の説明を区別する。確証のない意図は「〜と考えられる」等で書く。
- 構造・データフロー・状態遷移は Mermaid 図で可視化する（GitHub で描画される）。

## 7. 提出前の自己点検

- [ ] すべての引用にバージョン固定 GitHub リンクが付き、行番号が実ソースと一致している。
- [ ] ローカル絶対パスを書いていない。
- [ ] japanese-tech-writing 規範（一文一行、ダッシュ・中黒の不使用、空句排除）に従っている。
- [ ] 最適化の工夫を最低1つ機構レベルで説明した。
- [ ] Mermaid が描画可能な構文であり、章間リンクが実在ファイルを指している。
- [ ] 目次 `README.md` が §2 の体裁（`## 第N部`・末尾 `---`・箇条書き導入・一文一行）に従っている。
- [ ] `npx --yes markdownlint-cli2@0.22.1` をローカルで実行し、エラーが0件であることを確認した。

## 8. モデル運用（並列委譲する場合）

- 章執筆や重い読解はサブエージェントへ並列委譲してよい。委譲時は Agent / Workflow の `model` を明示する（実装・重い読解は `opus`、軽微な修正は `sonnet`）。
- メインループはオーケストレーション・引用リンクの正確性・日本語規範のレビューを担う。
- 1バッチ（1部など）ごとに §7 の点検を回し、次のバッチへ進む。

## 9. 新規 OSS の立ち上げ（自動ワークフロー）

ユーザーが OSS 名とソースコードパスを指示すれば、残りは自動で進める。
詳細手順は `.wiki/methodology.md` にある。以下が必須ゲートである。

- **AskUserQuestion（着手前）**: 規模感（章数）と GitHub リポジトリ/タグの2点を確認する。想定読者、図の方針、進め方はデフォルト値（中級エンジニア、Mermaid、直接執筆）を使い、聞かない。
- **タグ照合（着手前）**: WebFetch でタグの存在を確認し、ローカルソースと行番号を1ファイル突き合わせる。
- **検証ゲート（各部完了時）**: §7 の点検項目をバッチごとに機械的に実行し、パスしてから次の部へ進む。
- **仕上げ**: 全章完了後にエコシステム索引への追記、コミット、push、PR 作成まで一気に行う。
