# CLAUDE.md

このリポジトリ `source-code-reading` は、OSS のソースコード読解ドキュメントを日本語で蓄積する場である。
OSS ごとに専用ディレクトリ（例 `rocksdb/`）を持ち、その直下 `README.md` が目次となる。

## 執筆・改訂のルール

章を加筆・改訂するときは、必ずリポジトリルートの **`AGENTS.md`** を読み、その規約に従うこと。
要点（詳細は AGENTS.md）：

- 執筆前に `japanese-tech-writing` スキルを読み、その規範に従う。
- コード引用はバージョン固定の GitHub リンク（`.../blob/<tag>/...#Lxx-Lyy`）＋コードブロックの2点セット。行番号は実ソースを読んで厳密に確認する。ローカル絶対パスは書かない。
- 各章で最適化の工夫を最低1つ機構レベルで説明し、Mermaid 図で構造を可視化する。
- 用語は対象 OSS の既存章に合わせて統一する。

## 立ち上げ手順・背景知識

新しい OSS のドキュメントを始めるときの手順、過去の執筆履歴、再利用できる知見は `.wiki/`（`methodology.md`、`log.md`）にある。

## モデル運用

章執筆や重い読解はサブエージェントへ並列委譲してよい。委譲時は `model` を明示する（実装・重い読解は `opus`、軽微な修正は `sonnet`）。メインループはオーケストレーションと引用・規範のレビューを担う。

## 新規 OSS ドキュメントの開始

ユーザーが「次は〇〇をお願いします. ソースコードは〇〇です.」と指示した場合、その2点だけで着手する。
以下を自動で実行し、追加の指示を待たない。

1. `.wiki/methodology.md` を読み、全体の流れに従う。
2. ソースコードを探索し、主要サブシステムを把握する。
3. **AskUserQuestion** で以下の2点をユーザーに確認する（着手前に必ず）。
   - 規模感（コンパクト 10-12章 / 標準 15-18章 / 詳細 20章超）
   - GitHub リポジトリとタグ（go.mod 等から推定した候補を提示し確認）
4. 回答を得たら、以下をユーザーの追加指示なしに一気通貫で進める。
   - タグ存在確認（WebFetch）と行番号のスポット照合
   - 章構成の設計、README と執筆ガイドの作成
   - 部ごとの並列執筆（Opus サブエージェント）と検証ゲート（AGENTS.md §7）
   - 全部完了後にコミット、push、PR 作成
   - PR 作成後に `gh run list` で CI の実行を確認し、失敗していれば自動で修正してコミット、push する

以下はデフォルト値とし、ユーザーが明示しない限り確認しない。

- 想定読者: 対象言語と分散システムまたは DB の基礎がある中級エンジニア
- 図の方針: Mermaid
- 進め方: 目次承認なしで直接執筆
